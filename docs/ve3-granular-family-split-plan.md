# VE3 Granular Family Split — Implementation Plan

**Status:** Proposed. Next milestone after the literal conversion (T1–T10) reached zero across all 27 themes.
**Audience:** The engineer/agent implementing the per-family split end-to-end.
**Builds on:** [`docs/ve2-literal-conversion-plan.md`](./ve2-literal-conversion-plan.md) — the literal monoliths this plan partitions (see its §12 "Next milestone" note).

---

## 1. Goal

Divide each theme's verified literal monolith into **per-family style chunks** so the app loads only the families a route actually uses, **without reintroducing the heuristic family mapping** that the literal rewrite eliminated (the root cause of the mismatches the whole rewrite fixed).

The granular runtime, hooks, and taxonomy already exist (kept frozen by T10). This milestone produces the **data** they consume: per-family `styles.css.ts` files generated by partitioning the literal output, plus a regenerated loader map for all 27 themes.

## 3. The one real design problem

The literal rewrite's thesis (conversion plan §1, §25): heuristics caused the mismatches; replace them with **one complete deterministic 1:1 registry + census strictness**. The next-milestone note is explicit: do **not** revive the deleted heuristic family emitter.

But granular loading requires assigning every rule to a family — and the compiled `bootstrap.css` has **no section comments**, and `walkCssEmitUnits` in [`parse-css-tree.mjs`](../scripts/generate-ve-theme-literal/parse-css-tree.mjs) already discards comments and source positions. Family provenance does not exist at emit time. The old system reconstructed it with `guessFamilyForSelector` prefix-matching — exactly the heuristic the rewrite killed.

**Resolution: family assignment is not a rendering concern, so it does not need a rendering-grade heuristic — it needs the same treatment §5.3 gave elements: one complete, curated, census-verified `contract → family` table.**

This is the key reframing. The literal registry already maps every class 1:1 to a contract symbol, and most of those symbols already live in family-scoped contract dirs (`theme-contract/ui/buttons/contract.css`, etc.) — that directory structure *is* a near-complete family map already. Generate the `contract → family` table **once** from that structure, audit it to 0-unmapped with a census (like conversion-plan T1), and the partition becomes a deterministic lookup, not a guess.

> The table is to granular what the contract registry is to literal: the single auditable input that replaces all the per-family special cases.

## 4. Partitioning algorithm

For each emitted rule in the monolith (preserving its global source-order index `i`):

1. **Determine the subject family.** Take the rule's *rightmost compound segment* (the selector subject), look up its primary class contract in the `contract → family` table. `.navbar .nav-link {…}` → subject `.nav-link` → `ui/navs`. `.btn-primary {…}` → `ui/buttons`. Deterministic, and it matches how the existing composite route map is already built (the navbar route declares `[navbar, navs, dropdowns, buttons]` — its dependency closure).
2. **Reboot/element & `:root` var rules → `global`.** Bare-element rules (`ol ol`, `button{}`), `*{box-sizing}`, and the `${scope}${vars}` blocks go to the always-loaded `global` family. The body-split (`bodyFrame`/`bodyText`) lands here too.
3. **Emit each family file in ascending `i` order**, so within a family the cascade is identical to the monolith. Reuse the existing `buildImportBlock` logic per-family to pull only the contract symbols that family references.

### 4.1 Cross-family cascade (the second and only other real risk)

In the monolith, two equal-specificity rules resolve by global source order. Split into separately-injected chunks, resolution flips to *load order*, which is nondeterministic. Two defenses, preferred first:

- **Wrap every family chunk in a single shared `@layer`** ordered by each family's first-appearance source index. VE supports layers; this makes cross-family cascade deterministic regardless of dynamic-import order. **Lead with this.**
- Failing that, lean on conversion-plan §10.2 (equal-specificity cross-family conflicts are ~5%, surfaced case-by-case) and carry the global `i` index so any conflict is diagnosable.

### 4.2 Compound-rule ownership tie-breaks

Subject-family assignment is clean for single-subject rules. Define one explicit rule (in the table, not in code branches) for the ambiguous cases:
- Grouped selectors spanning families (`body, input, button {…}`) → assign to `global`, **or** split the group by subject.
- Universal/structural subjects (`.table > :not(caption) > * > *`) → assign by the nearest *named* segment to the left of the structural tail.

This is the single place ambiguity can creep in — pin it down once, explicitly.

## 5. The new completeness invariant (granular analog of Phase 3)

The monolith cannot under-load — everything is always present. Granular introduces one new failure mode: **a route renders an element whose matching rule lives in a family the route never requested.** This is exactly what the 342 `useVe2RequiredStyleFamilies` declarations + `route-style-families.ts` must cover.

**Closure check** (built on [`markup-class-parity.mjs`](../scripts/capture-leaf-screenshots/markup-class-parity.mjs)): for each scenario, map every contract its DOM stamps → its family (via the new table) → assert that family is in the union the scenario's components declare. Any gap is a missing `ve2RequiredStyleFamilies` entry, reported deterministically before any screenshot runs.

The pixel gate is then **free**: `--style-loader=granular` already exists. Run the same 433-scenario `verify-ve-family` loop in granular mode. Because each family chunk is byte-identical rules to the monolith in the same relative order, a correctly-closed route verifies to zero with **no per-family CSS fixes** — only closure/order fixes.

---

## 6. Where it hooks in

The partition sits **after** `walkCssEmitUnits`, **before** per-file write, in [`emit.mjs`](../scripts/generate-ve-literal/emit.mjs) (around the main loop, conversion-plan refs the `bodyLines[]` accumulator at emit.mjs:589): change the single `bodyLines[]` accumulator into a `Map<family, lines[]>` keyed by the table lookup, and write one file per family instead of one monolith (optionally keep emitting the monolith too, for the literal loader).

## 7. Step-by-step task list

Mirrors the conversion plan's task style. Each task has an acceptance gate; do not advance until it passes.

**✅ G1 — Family table + census.** Generate `contract → family` from the existing `theme-contract/{family}/` dirs; add `--census` to report 0-unmapped across 27 themes. (Mirrors conversion-plan T1.)
*Gate:* census runs; every class contract maps to exactly one family (documented tie-breaks aside); 0 unmapped.

**Done.** [`scripts/generate-ve-literal/family-table.mjs`](../scripts/generate-ve-literal/family-table.mjs) created. `buildFamilyTable()` resolves each contract symbol → emit family by reusing `buildContractRegistry().contractsByFamily` under the same priority order as the emitter's `buildElementImportPaths` (specific family > utilities > generated; **`literal` excluded** so literal-only symbols surface as gaps). `contractFamilyToStyleFamily()` folds the contract-dir namespaces onto the `VE2_STYLE_FAMILIES` taxonomy (`utilities/generated`→`utilities`; `global-elements`/`root-contract`/`layout`→`global`). Element/reboot contracts (`TAG_TO_CONTRACT`) pinned to `global`.

Census (`--census --all-themes`): **2069 class symbols, 0 unmapped, 0 invalid** across all 27 themes, via **105 `FAMILY_OVERRIDES`** + the dir structure.

**Audit finding (utilities sanity check).** The first pass put **1796/2069 (86.8%)** symbols in `utilities`, which looked suspiciously high. Investigation confirmed the *magnitude* is legitimate — 1754 come from Bootstrap's combinatorial utility API (`utilities/generated/contract.css`: the `m/p-{side}-{bp}-{0..5}` spacing matrix, `align-*`, `text-*`, `w-*`, `z-*`, etc.) and **0 symbols were wrongly stolen from a specific family** (priority order intact). **But** the inherited `utilities/generated` catch-all had swept in **65 genuine component classes** (`.modal-lg`, `.modal-fullscreen-*`, `.dropdown-menu-{bp}-{start,end}`, `.navbar-expand*`, `.list-group-horizontal*`, `.btn-group-vertical`, `.valid-tooltip`, `.accordion-flush`, `.toast-container`, …) — exactly the "catch-all dumping" this rewrite removes. **51** were reclassified to their real family via overrides (semantically correct + avoids dragging `.modal-lg` onto every route). The remaining **14 `.offcanvas*`** are left in `utilities` as **dead CSS** — there is no `offcanvas` family in the taxonomy and no offcanvas component/scenario exists. Final utilities count: **1745**.

The override table (105 entries) covers: literal-only legacy/Bootswatch classes (Bootswatch card variants → `ui/card`; `.h1`–`.h6` → `contents/heading`; BS3 form/navbar/table shims → their families; Tether/popover positioning → `ui/popovers`; `.rounded-pill` → `utilities`; sandstone's grouped `.sandstone` → `global`), plus the 51 catch-all reclassifications above. Lint clean (Biome).

**G2 — Partition emitter.** Map-keyed accumulator + per-family `buildImportBlock` + `@layer` ordering; emit `themes/{theme}/{family}/styles.css.ts`. (Mirrors T4.)
*Gate:* bootstrap emits all families with 0 skipped; union of family files is rule-equivalent to the monolith.

**G3 — Regenerate loader map.** Regenerate `ve2ThemeFamilyLoaders` for all **27** themes; expand `Ve2GranularShell` theme resolution beyond the current 8. (Mirrors T5.)
*Gate:* `?style-loader=granular&theme=<t>` loads each theme's family chunks for every theme.

**G4 — Closure check.** Family-coverage assertion on top of markup-parity; fix `ve2RequiredStyleFamilies` gaps. (Mirrors Phase 3.)
*Gate:* closure report empty for the families being verified.

**G5 — Verify to zero.** `--style-loader=granular`, all 27 themes; fixes go to the table or the closure declarations, **never per-family CSS**. (Mirrors T8/T9.)
*Gate:* all 27 themes, all families, 0 over `0.001` in granular mode.

## 8. Open risks

1. **Compound-rule ownership** (§4.2) — the one place ambiguity can creep in; define the tie-break in the table, explicitly, once.
2. **The 342 declarations are stale.** They were authored against the *heuristic* split; some will be wrong against the new deterministic table. G4's closure check catches this — budget for declaration churn.
3. **`@layer` interaction with scope/var injection.** Confirm the `${scope}${vars}` blocks and `fonts.generated.css` ordering still win correctly once everything is layered.
4. **Theme reach.** Loader map + granular shell cover only 8 themes today; G3 must extend both to all 27 (same scope expansion T5 did for the literal loader).

---

## 9. Why this keeps faith with the rewrite

The milestone reduces to **one deterministic table + a partition step + a closure check**. Family assignment becomes a curated registry problem (like the §5.3 element table), not a guessing problem — so granular loading is layered on top of the verified literal foundation without reintroducing the heuristic mapping the rewrite was built to remove.
