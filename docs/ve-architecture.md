# Vanilla Extract Architecture (Element-Owned Scope)

This document defines the current Vanilla Extract (VE) architecture for this repository.
All new Bootstrap-to-VE component conversions should follow this approach.

Implementation: [`ve-project2/`](../ve-project2)

Companion documents:

- [`docs/ve-architecture-v1-history.md`](./ve-architecture-v1-history.md) — archived architecture of the first VE sub-project (`ve-project`), preserved as historical context.
- [`docs/ve-project1-docs-index.md`](./ve-project1-docs-index.md) — index of all historical docs for ve-project (v1).

## Core Philosophy

The element-owned scope model removes two problems that plagued `ve-project` (v1):

1. **No global element styling.** Nothing styles bare elements (`body`, `h1`, `button`, etc.) globally. Every visual rule is opt-in, applied through explicit class composition.
2. **No ancestor/descendant CSS matching.** Theme styles do not rely on a parent carrying a scope class and descendants inheriting from it. Every styled element carries **both** the scope class and the contract class directly, producing a compound selector with no space in between (`.scope.contract`).

This means unlimited nesting depth, zero specificity conflicts, and full theme overridability without `@layer` or extra selectors.

## Concepts

### Scope class

One empty `style({})` per theme. The class name is a stable hashed identifier generated by Vanilla Extract. It acts as a **namespace prefix** for all CSS rules belonging to that theme.

```ts
// ve-project2/src/themes/bootstrap/scope.css.ts
export const bootstrapScope = style({})
```

No CSS properties are embedded in the scope class itself. It is a zero-style identifier.

### Contract class

One empty `style({})` per component slot (button base, button primary variant, body, bodyText, etc.). Contract classes live in `theme-contract/` and are shared by all themes.

```ts
// ve-project2/src/theme-contract/ui/buttons/contract.css.ts
export const btn = style({})
export const btnPrimary = style({})
// …
```

```ts
// ve-project2/src/theme-contract/theme-contract.css.ts
export const body = style({})     // outer wrapper: background, padding, display
export const bodyText = style({}) // text root: font-family, font-size, color
```

Again, no CSS properties embedded — only stable hashed class names.

### Theme rules

All visual styling lives in `globalStyle` calls that target a **compound selector** (`scope + contract` with no space). This means the rule fires only on an element that carries **both** classes simultaneously.

```ts
// ve-project2/src/themes/bootstrap/ui/buttons/styles.css.ts
import { globalStyle } from '@vanilla-extract/css'
import { btn, btnPrimary } from '../../../../theme-contract/ui/buttons/contract.css'
import { bootstrapScope } from '../../scope.css'

globalStyle(`${bootstrapScope}${btn}`, {
  display: 'inline-block',
  padding: '0.375rem 0.75rem',
  // …
})

globalStyle(`${bootstrapScope}${btnPrimary}`, {
  color: '#fff',
  backgroundColor: '#0d6efd',
  borderColor: '#0d6efd',
})
```

Root/body-level rules follow the same pattern, using `body`/`bodyText` contract classes:

```ts
// ve-project2/src/themes/bootstrap/scope.css.ts
import { globalStyle, style } from '@vanilla-extract/css'
import { body, bodyText } from '../../theme-contract/theme-contract.css'

export const bootstrapScope = style({})

globalStyle(`${bootstrapScope}${bodyText}`, {
  fontFamily: 'system-ui, -apple-system, "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif',
  fontSize: '1rem',
  fontWeight: 400,
  lineHeight: 1.5,
  color: '#212529',
})

globalStyle(`${bootstrapScope}${body}`, {
  backgroundColor: '#fff',
  padding: '1rem',
  display: 'block',
})
```

### ThemeContext

A SolidJS context that propagates the active scope class string to every descendant component. Components read it and stamp it on every element they render alongside their contract classes.

```ts
// ve-project2/src/context/ThemeContext.ts
import { createContext } from 'solid-js'
export const ThemeContext = createContext('')
```

### Element stamping (components)

Every rendered element carries both the scope class (from context) and its contract class(es):

```tsx
// ve-project2/src/components/ui/buttons/solid/SolidPrimaryButton.tsx
const theme = useContext(ThemeContext)

<button type="button" class={`${theme} ${btn} ${btnPrimary} pwhook-btn`}>
  Primary
</button>
```

The resulting CSS selector is `.bootstrapScope_xyz.btn_abc { … }` — no ancestor matching, no inheritance dependency.

### ThemedBody

A small wrapper component that stamps `scope + body + bodyText` on a `<div>`, activating the root-level typographic and layout rules for a themed subtree.

```tsx
const ThemedBody: Component<{ children: any }> = (props) => {
  const theme = useContext(ThemeContext)
  return <div class={`${theme} ${body} ${bodyText}`}>{props.children}</div>
}
```

Use `ThemedBody` at the root of any themed region and whenever the theme switches in nested `ThemeContext.Provider` blocks.

### Ve2Shell

The application shell for `ve-project2`. Reads the `?theme=` URL parameter, resolves the matching scope class, provides it via `ThemeContext`, and wraps content in a `ThemedBody`-equivalent `<div>`.

```ts
// ve-project2/src/components/shell/Ve2Shell.tsx
const themeClass = () => resolveThemeClass(params.get('theme'))

<ThemeContext.Provider value={themeClass()}>
  <div class={`${themeClass()} ${body} ${bodyText}`}>
    {props.children}
  </div>
</ThemeContext.Provider>
```

## Core Rules

1. **Scope and contract classes must be empty `style({})`** — no CSS properties embedded.
2. **All styling goes through `globalStyle(scope + contract, props)`** — compound selector with no space.
3. **No global element selectors** (`body`, `button`, `h1`, etc.).
4. **No `createVar()` or CSS custom properties** needed for basic component styling in this model.
5. **Every styled element carries both the scope class and its contract class(es).**
6. **Body/text styles use `body`/`bodyText` contract classes**, not the scope class directly.
7. **Only these static class names are allowed in component markup:** `bd-example-ve2`, `pwhook-*` screenshot hooks. All other styling comes from VE imports.
8. **ThemeContext provides the active scope class string.** Components read it via `useContext(ThemeContext)` — no prop-drilling.
9. **`ThemedBody` (or equivalent) must wrap any region that requires body-level theme styles** (font, color, background).

## Adding a New Bootstrap Component Family

Use this checklist for every new component family (e.g. inputs, badges, modals).

### 1. Define contract classes

Create `ve-project2/src/theme-contract/ui/{family}/contract.css.ts`:

```ts
import { style } from '@vanilla-extract/css'

export const myComponent = style({})
export const myComponentVariant = style({})
// …
```

All exports must be `style({})` — no properties.

### 2. Implement theme rules

For each theme that implements the family, create `ve-project2/src/themes/{theme}/ui/{family}/styles.css.ts`:

```ts
import { globalStyle } from '@vanilla-extract/css'
import { myComponent, myComponentVariant } from '../../../../theme-contract/ui/{family}/contract.css'
import { bootstrapScope } from '../../scope.css'

globalStyle(`${bootstrapScope}${myComponent}`, { /* base styles */ })
globalStyle(`${bootstrapScope}${myComponentVariant}`, { /* variant styles */ })
```

Import the styles file as a **side-effect import** in the component or shell that uses it:

```ts
import 'path/to/styles.css'
```

### 3. Create components

Create component files in `ve-project2/src/components/ui/{family}/`:

```tsx
import { useContext } from 'solid-js'
import { ThemeContext } from '../../../../context/ThemeContext'
import { myComponent, myComponentVariant } from '../../../../theme-contract/ui/{family}/contract.css'

const MyComponent = () => {
  const theme = useContext(ThemeContext)
  return <div class={`${theme} ${myComponent} ${myComponentVariant}`}>…</div>
}
```

### 4. Wire routes

Add explicit `<Route>` entries in `ve-project2/src/index.tsx` for each leaf component.

### 5. Add theme side-effect import

In `Ve2Shell.tsx` (or the relevant shell), add:

```ts
import '../../themes/bootstrap/ui/{family}/styles.css'
import '../../themes/sketchy/ui/{family}/styles.css'
```

## Multi-Theme Nesting

Because every element carries its own scope class, multiple themes can be nested at any depth without conflicts. Each `ThemeContext.Provider` overrides the scope for its subtree:

```tsx
<ThemeContext.Provider value={bootstrapScope}>
  <ThemedBody>
    <Button />  {/* Bootstrap-styled */}
    <ThemeContext.Provider value={sketchyScope}>
      <ThemedBody>
        <Button />  {/* Sketchy-styled, no conflicts */}
      </ThemedBody>
    </ThemeContext.Provider>
  </ThemedBody>
</ThemeContext.Provider>
```

No `@layer`, no specificity tricks, no extra selectors needed for any nesting depth.

## Cross-Class Selector Rule

For patterns that depend on sibling or adjacent-sibling relationships (e.g. Bootstrap's `btn-check` + adjacent button label), use `globalStyle` with the scope prefix and both generated class references:

```ts
// Disabled checkbox makes the adjacent button label look disabled
globalStyle(`${bootstrapScope}${btnCheck}:disabled + ${bootstrapScope}${btn}`, {
  pointerEvents: 'none',
  opacity: '0.65',
})
```

Do **not** use raw class name strings (e.g. `'.btn'`) in selectors. Always reference the generated VE identifiers.

## Historical Context

The first VE sub-project (`ve-project`) used a different approach:

- CSS custom properties (`createVar()`) for theme tokens
- Runtime theme resolver and typed family contracts
- URL-driven theme selection with skip-with-warning for partial coverage
- Scope class carried CSS properties directly (not empty)

That approach is fully documented in:

- [`docs/ve-architecture-v1-history.md`](./ve-architecture-v1-history.md)
- [`docs/ve-project1-docs-index.md`](./ve-project1-docs-index.md)

The element-owned scope model in `ve-project2` was chosen because it eliminates the runtime contract complexity while delivering better multi-theme nesting support and a simpler CSS output.
